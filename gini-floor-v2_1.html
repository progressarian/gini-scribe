Role-based views — do you need separate dashboards?
Short answer: one shared database, role-filtered views. Not entirely separate apps.
The answer — one database, filtered by role
You don't need to build 6 separate dashboards. You need one data model with role-based filters applied. Every role sees the same patient journey data — just filtered to what is relevant to them and what actions they can take. The coordinator sees everything. The vitals staff sees only the vitals step. The SD sees only their assigned patients. Dr. Bhansali sees only Red and Amber patients who have completed the SD step.

How the SD handles "tests first — then Dr. Bhansali"
When the SD sees a patient with no reports (or after prescribing tests), they have two options to record in Scribe:
Option A — Same-day results
SD orders tests → patient goes to lab → results land in Scribe → patient automatically re-enters the queue marked "Results received — ready for Dr. Bhansali." The floor dashboard shows them as "Awaiting lab results" with estimated lab turnaround time visible to the coordinator.
Option B — Next visit with results
SD records the visit, orders tests, and marks "Next appointment — bring results." A new appointment is created in Scribe pre-tagged with the required tests. When the patient books, the coordinator can see the pending test list and chase them up before the next visit via MHG or phone call.
For Ortho and Pulmo — same structure, no separate build
The department dropdown in the triage and floor dashboard controls which patients are shown. For a single-doctor clinic (Dr. Harjoban in Ortho) there is no SD layer — the patient goes directly from vitals to the doctor. The triage column becomes one column (the doctor's queue) instead of three. The role filter then shows "Dr. Harjoban" patients only. One codebase with department and role filters covers all three specialties.

Developer brief — what to build
Role-based login
Add a role field to the existing doctor/staff login. Roles: coordinator, vitals_staff, specialist_doctor, senior_doctor, billing, pharmacy. Each role gets a filtered view of the same appointment data — not a separate app. The existing PIN login can be extended with a role dropdown.
New status values
Extend appointment status to: booked → confirmed_call → confirmed_day_before → checked_in → vitals_in_progress → vitals_done → waiting_sd → with_sd → awaiting_lab → results_received → waiting_dr_bhansali → with_dr_bhansali → prescription_pending → billing → pharmacy → completed. Each status transition also writes a timestamp field.
Who sees what
Vitals staff: only appointments in status checked_in or vitals_in_progress. SD: only their assigned appointments in vitals_done, waiting_sd, with_sd. Dr. Bhansali: Red+Amber in results_received or waiting_dr_bhansali. Coordinator: all. Auto-advance status when SD taps "ready for Dr. Bhansali."
Department filter
Add department field to appointments (diabetes, ortho, pulmo). The floor dashboard and triage view filter by selected department. The triage column count adjusts — 3 columns for diabetes (multi-doctor), 1 column for single-doctor departments. Same codebase, department config determines the layout.
